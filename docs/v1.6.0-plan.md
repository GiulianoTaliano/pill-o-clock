### 1. AM/PM format in time controls

**Context:**
Currently all `DateTimePicker` components in the app use `is24Hour={true}`,
forcing 24h format regardless of the user's system preference. Many Spanish-speaking
and English-speaking users prefer 12h format (AM/PM).

**Affected files:**
- `components/MedicationForm.tsx` — `ScheduleRow` (alarm time picker)
- `app/(tabs)/appointments.tsx` — appointment time picker
- `app/(tabs)/index.tsx` — dose reschedule picker (if using DateTimePicker)

**Implementation plan:**

1. **Remove `is24Hour={true}`** from all `DateTimePicker` instances.
   By default, the native picker inherits the device setting (12h or 24h).
   No additional prop is needed — removing the override is sufficient.

2. **Display times consistently:**
   The app currently shows times as raw strings (`"08:00"`, `"20:30"`). When
   displaying a time stored as `"HH:mm"`, use a utility that formats it
   according to the device locale (12h or 24h), and store it always as `"HH:mm"`.

   Add a `formatTimeForDisplay(time: string): string` utility to `src/utils/`:

   ```ts
   // src/utils/time.ts
   import { format, parse } from "date-fns";

   /**
    * Formats a stored "HH:mm" time string for display according to
    * the device locale (12h with AM/PM or 24h).
    */
   export function formatTimeForDisplay(time: string): string {
     const date = parse(time, "HH:mm", new Date());
     // Intl.DateTimeFormat respects system locale (12h or 24h)
     return new Intl.DateTimeFormat(undefined, {
       hour: "numeric",
       minute: "2-digit",
       hour12: undefined, // let the system decide
     }).format(date);
   }
   ```

3. **Apply the utility** in all places that display a stored `"HH:mm"` time:
   - `DoseCard.tsx` — time badge
   - `MedicationCard.tsx` — next-dose line
   - `MedicationForm.tsx` — `ScheduleRow` time display
   - `app/(tabs)/appointments.tsx` — appointment time display

**Acceptance criteria:**
- [ ] All `DateTimePicker` components have `is24Hour` removed
- [ ] On a device set to 12h, times display as "8:00 AM", "8:30 PM"
- [ ] On a device set to 24h, times display as "08:00", "20:30"
- [ ] Stored values remain as `"HH:mm"` (no format change in DB or store)
- [ ] `formatTimeForDisplay()` has unit tests for both 12h and 24h cases

### 2. Document attachments on appointments

**Context:**
Appointments currently only store text data (place, date, time, notes, location).
Users need to attach documents to appointments: lab results, referral letters,
prescriptions, etc. These documents should be viewable directly from the
appointment detail.

**Affected files:**
- `src/db/schema.ts` — new `appointment_documents` table
- `src/store/slices/appointments.ts` — new CRUD actions for documents
- `src/store/types.ts` — `AppointmentDocument` type + slice actions
- `src/types/index.ts` — domain interface `AppointmentDocument`
- `components/AppointmentDetailModal.tsx` — document list + attach/open actions
- `src/services/backup.ts` — include documents in export/import

**Implementation plan:**

1. **New DB table — `appointment_documents`:**

   ```ts
   // src/db/schema.ts
   export const appointmentDocuments = sqliteTable(
     "appointment_documents",
     {
       id: text("id").primaryKey(),
       appointmentId: text("appointment_id")
         .notNull()
         .references(() => appointments.id, { onDelete: "cascade" }),
       fileName: text("file_name").notNull(),
       mimeType: text("mime_type").notNull(),
       fileUri: text("file_uri").notNull(),   // local path in documentDirectory
       fileSize: integer("file_size"),        // bytes, optional
       createdAt: text("created_at").notNull(),
     },
     (t) => [index("idx_appointment_docs_appointment_id").on(t.appointmentId)]
   );
   ```

2. **DB migration — v8 → v9:**

   ```ts
   // src/db/database.ts — inside migrateDatabase()
   case 8:
     await db.run(sql`
       CREATE TABLE IF NOT EXISTS appointment_documents (
         id               TEXT PRIMARY KEY,
         appointment_id   TEXT NOT NULL REFERENCES appointments(id) ON DELETE CASCADE,
         file_name        TEXT NOT NULL,
         mime_type        TEXT NOT NULL,
         file_uri         TEXT NOT NULL,
         file_size        INTEGER,
         created_at       TEXT NOT NULL
       )
     `);
     await db.run(sql`
       CREATE INDEX IF NOT EXISTS idx_appointment_docs_appointment_id
         ON appointment_documents(appointment_id)
     `);
     break;
   ```

3. **TypeScript type:**

   ```ts
   // src/types/index.ts
   export interface AppointmentDocument {
     id: string;
     appointmentId: string;
     fileName: string;
     mimeType: string;
     fileUri: string; // path inside FileSystem.documentDirectory
     fileSize?: number;
     createdAt: string;
   }
   ```

4. **File storage:**
   When the user picks a document, copy it to `FileSystem.documentDirectory +
   "appointment_docs/"` with a unique name (UUID + original extension). This ensures
   the file persists between app sessions and is independent of the system's
   temporary directory.

5. **Store slice actions (`appointments.ts`):**
   - `addAppointmentDocument(appointmentId, pickerResult)` — copy file + insert row
   - `removeAppointmentDocument(docId)` — delete file + delete row
   - `loadAppointmentDocuments(appointmentId)` — fetch all documents for the appointment

6. **UI — `AppointmentDetailModal.tsx`:**
   - New "Documents" section below the notes field.
   - "Attach document" button → `expo-document-picker` (allows PDF, images, DOC, etc.).
   - Each document shows: file name, size, date, delete icon.
   - Tapping a document opens it with `expo-sharing` / `Sharing.shareAsync()`,
     which on Android uses the native intent to open with the appropriate app
     (PDF viewer, Gallery, etc.).

7. **Native viewer strategy:**
   - `expo-sharing` / `Sharing.shareAsync(fileUri, { mimeType })` triggers the
     native "open with / share" dialog. On Android this covers > 95% of use cases
     (PDF, images, DOC). It does not require implementing a custom viewer.
   - Alternative: `Linking.openURL("file://...")` for direct opening — may fail
     if the file provider is not configured. `expo-sharing` is safer.

8. **Backup:**
   - The `fileUri` column stores a relative path inside `documentDirectory`.
   - Backup includes the documents table but NOT the binary files (they are local
     device files). The roadmap for v1.7+ may include binary export (zip).

**Acceptance criteria:**
- [ ] The user can attach one or more documents to an appointment
- [ ] Attached documents are listed in the appointment detail with name and date
- [ ] Tapping a document opens the native viewer/sharing dialog
- [ ] Documents are deleted when the appointment is deleted (ON DELETE CASCADE)
- [ ] The user can manually delete an individual document (file + row)
- [ ] Fresh install and migration from v1.5.0 work correctly
- [ ] The document file survives app restart (stored in `documentDirectory`)

### 3. Stop alarm when the notification is opened or dismissed

**Context:**
When the alarm triggers, Android shows a fullscreen notification that opens
`app/alarm.tsx`. Currently, the alarm audio (native `AlarmAudioService.kt`)
continues playing even after the user opens the alarm screen — it only stops when
the user taps "Confirm" or "Snooze". The audio should stop as soon as:
- (a) The user opens the alarm screen (swipes up / taps the notification)
- (b) The user swipes away / dismisses the notification without opening it

**Affected files:**
- `app/alarm.tsx` — add `stopAlarm()` call on mount
- `modules/expo-alarm/android/.../AlarmAudioService.kt` — already handles case (b) ✅

**Implementation plan:**

1. **Case (b) — swipe dismiss:**
   Already handled: `AlarmAudioService.kt` has a `dismissReceiver` that listens
   for `ACTION_DISMISS` and calls `stopSelf()`, which stops the `MediaPlayer`.
   No changes needed in Kotlin.

2. **Case (a) — open the alarm screen:**
   Add `stopAlarm()` inside the existing `useEffect` in `alarm.tsx` that already
   calls `setAlarmWindowFlags()`:

   ```tsx
   // app/alarm.tsx
   useEffect(() => {
     setAlarmWindowFlags(true);
     stopAlarm().catch(() => {}); // stop audio as soon as the screen mounts
     return () => {
       setAlarmWindowFlags(false);
     };
   }, []);
   ```

   `stopAlarm()` is already imported and available in `alarm.tsx`. Adding it
   here ensures that opening the alarm screen from the notification immediately
   stops the alarm sound, even before the user taps any button.

**Acceptance criteria:**
- [ ] Audio stops immediately when the alarm screen opens
- [ ] Audio stops when the user swipes away the notification (existing behavior preserved)
- [ ] The "Confirm" and "Snooze" buttons continue to work correctly
- [ ] No double `stopAlarm()` call causes errors (idempotent)

— "I'll be out of the bathroom in a moment") or longer (30–60 min — "I'm

alarm screen and in the DoseCard, defaulting to 15 minutes.

**Affected files:**

### 7. Differentiated time badge background in DoseCard

**Context:**
According to the provided screenshots, the time badge in `DoseCard` uses
`colors.light` as its background — a very light shade of the medication color.
In both light and dark mode, this background blends with the card background
(which uses `statusTheme.bg`). The badge needs more contrast to stand out
from the card background.

**Affected files:**
- `components/DoseCard.tsx` — time badge styles

**Implementation plan:**

1. **Identify the current state:**
   The badge uses `backgroundColor: colors.light` and `borderColor: colors.border`
   when not snoozed. For most colors, `colors.light` is a very soft pastel
   (e.g. light green `#dcfce7`), similar to the card background.

2. **Solution — use a more saturated background for the badge:**
   Increase the opacity of the medication color in the badge background, or use
   an intermediate color that creates more visual contrast.

   ```tsx
   // DoseCard.tsx — time badge (when not snoozed)
   style={{
     backgroundColor: isSnoozed
       ? (theme.isDark ? "#451a03" : "#fef3c7")
       : (theme.isDark ? colors.bg + "30" : colors.bg + "20"),
     // Before: colors.light (very low contrast with the card)
     // After: medication color with low opacity (more visible)
     borderColor: isSnoozed
       ? (theme.isDark ? "#92400e" : "#fcd34d")
       : colors.bg + "55",
     // Border uses the medication color with medium opacity to reinforce it
   }}
   ```

   More robust alternative: use `colors.bg` with opacity `0x33` for the background
   and `0x77` for the border, creating a badge that is clearly "in the medication's
   color" without blending into the card background.

3. **Verify across all medication colors:**
   Visually test with the 7 preset colors (blue, green, purple, orange,
   red, teal, pink) and at least one custom hex color, in both modes (light/dark).

**Acceptance criteria:**
- [ ] The time badge is clearly distinct from the card background in light mode
- [ ] The time badge is clearly distinct from the card background in dark mode
- [ ] Works with all preset and custom medication colors
- [ ] The snoozed badge (amber) is not affected
- [ ] WCAG AA: badge text remains readable (contrast ≥ 4.5:1)

### 8. Medication form as a multi-slide wizard

**Context:**
The current "New Medication" form (`MedicationForm.tsx`) is a long vertical scroll
with all sections visible simultaneously. This works, but for the target audience
(elderly users) it is overwhelming: opening the screen presents a number of fields
that can be intimidating before they even start.

The redesign converts the form into a **horizontal slide wizard** (step by step)
where each slide focuses on a logical group of fields. The user advances by swiping
horizontally or tapping "Next", and can go back. This reduces the cognitive load
per screen and guides the user step by step.

**Analysis of the current form:**

The current form consists of 6 sections, controlled by `react-hook-form`
+ `zod` validation. Below is a full field inventory:

| Current section | Fields | Required | Notes |
|-----------------|--------|----------|-------|
| Medication info | `name` (text) | **Yes** | Medication name |
| | `dosageAmount` (number) | **Yes** | Numeric quantity |
| | `dosageUnit` (chips) | **Yes** | mg, g, mcg, ml, drops, tab., cap., IU |
| | `category` (chips) | **Yes** | Antibiotic, analgesic, anti-inflammatory, supplement, vitamin, other. Default: "other" |
| | `notes` (multiline text) | No | Instructions / free notes |
| | `color` (color picker) | **Yes** | Color for visual identification. Default: "blue" |
| Photo | `photoUri` (image picker) | No | Photo of the box/blister |
| Frequency | `repeatMode` (3 options) | **Yes** | "Once" / "Repeat" / "On demand (PRN)". Default: "repeat" |
| When / Period | `onceDate` or `startDate`/`endDate` (date pickers) | Partial | `onceDate` required if mode=once; `startDate`/`endDate` optional if mode=repeat |
| Alarms | `schedules[]` — each: `time` (HH:mm) + `days[]` (0-6) | **Yes** (if not PRN) | Minimum 1 alarm. Multiple alarms possible in mode=repeat |
| Stock | `stockQuantity` (number) + `stockAlertThreshold` (number) | No | Only if the user wants stock tracking |

**Affected files:**
- `components/MedicationForm.tsx` — complete render restructure
- `app/medication/new.tsx` — no functional changes (consumes `MedicationForm`)
- `app/medication/[id].tsx` — no functional changes (consumes `MedicationForm`)
- `src/schemas/medication.ts` — no changes (validation stays the same)
- `src/i18n/es.ts` / `src/i18n/en.ts` — new strings for wizard (step indicators, navigation)

**Implementation plan:**

#### 8.1. Slide design

Divide the form into **5 slides** that group fields by logical context.
Each slide has a title, a descriptive subtitle, and a limited set of fields
to reduce cognitive load.

| Slide | Title | Fields | Can be skipped? | Reason |
|-------|-------|--------|-----------------|--------|
| **1. Identity** | "What medication is it?" | `name`*, `dosageAmount`*, `dosageUnit`*, `category` | **No** | Name and dosage are the minimum data. Category has default "other" so the user can accept without touching it. |
| **2. Appearance** | "Customize your medication" | `color`, `photoUri` | **Yes** — "Skip" button | Color has a default (blue), photo is optional. The user may not need to customize. |
| **3. Frequency** | "How often do you take it?" | `repeatMode`, `onceDate` (if once), `startDate`/`endDate` (if repeat) | **No** | The frequency type determines the rest of the flow. Default "repeat" allows fast progress. |
| **4. Alarms** | "When should we remind you?" | `schedules[]` (time + days) | **No** (if not PRN) / **Auto-skip** (if PRN) | Without an alarm there is no reminder. If the user chose PRN in slide 3, this slide is automatically skipped. |
| **5. Extras** | "Additional details" | `notes`, `stockQuantity`, `stockAlertThreshold` | **Yes** — "Skip and save" button | All fields are optional. Allow saving directly without filling anything. |

#### 8.2. Wizard component — architecture

The wizard is implemented as a horizontal `FlatList` with paging enabled,
wrapped in the same `react-hook-form` context that already exists. There is no
change in validation logic or the schema — only the UI is redistributed.

```tsx
// Conceptual structure
function MedicationForm({ ... }) {
  const flatListRef = useRef<FlatList>(null);
  const [currentSlide, setCurrentSlide] = useState(0);
  const totalSlides = repeatMode === "prn" ? 4 : 5; // skip slide 4 if PRN

  // react-hook-form setup (unchanged)
  const { control, handleSubmit, watch, setValue, formState } = useForm(...);

  const slides = [
    <SlideIdentity />,
    <SlideAppearance />,
    <SlideFrequency />,
    ...(repeatMode !== "prn" ? [<SlideAlarms />] : []),
    <SlideExtras />,
  ];

  function goNext() {
    // Validate current slide fields before advancing
    if (currentSlide < totalSlides - 1) {
      flatListRef.current?.scrollToIndex({ index: currentSlide + 1 });
      setCurrentSlide(currentSlide + 1);
    }
  }

  function goBack() {
    if (currentSlide > 0) {
      flatListRef.current?.scrollToIndex({ index: currentSlide - 1 });
      setCurrentSlide(currentSlide - 1);
    }
  }

  return (
    <View className="flex-1 bg-background">
      {/* Progress indicator */}
      <StepIndicator current={currentSlide} total={totalSlides} />

      {/* Slides */}
      <FlatList
        ref={flatListRef}
        data={slides}
        horizontal
        pagingEnabled
        scrollEnabled={false} // only navigate via buttons/swipe gesture
        showsHorizontalScrollIndicator={false}
        renderItem={({ item }) => (
          <View style={{ width: screenWidth }}>{item}</View>
        )}
      />

      {/* Navigation footer */}
      <WizardFooter
        currentSlide={currentSlide}
        totalSlides={totalSlides}
        onBack={goBack}
        onNext={goNext}
        onSkip={...}
        onSubmit={handleSubmit(...)}
        isLastSlide={currentSlide === totalSlides - 1}
        isSubmitting={isSubmitting}
        canSkip={isSkippableSlide(currentSlide)}
      />
    </View>
  );
}
```

#### 8.3. Navigation and gestures

- **Horizontal swipe:** The `FlatList` with `pagingEnabled` allows free swiping
  between slides. However, swiping forward is only allowed if the required fields
  of the current slide are complete. If not, show a toast indicating what is missing.
- **"Next" button:** Validates current slide fields before advancing.
- **"Back" button:** Always enabled (no validation when going back).
- **"Skip" button:** Only visible on slides marked as skippable (2 and 5).
  Advances to the next slide leaving defaults.
- **"Save" / "Add medication" button:** Only appears on the last slide,
  replacing "Next". Runs a full schema validation.

#### 8.4. Progress indicator

A visual indicator at the top showing progress:

```
  ●───●───○───○───○
  1   2   3   4   5
```

- Filled dot + colored line = completed
- Filled dot with ring = current
- Empty dot = pending
- Color: `primary` (blue) for completed/current, `border` for pending

`StepIndicator` component:
```tsx
interface StepIndicatorProps {
  current: number;
  total: number;
}

function StepIndicator({ current, total }: StepIndicatorProps) {
  return (
    <View className="flex-row items-center justify-center px-8 py-4 gap-1">
      {Array.from({ length: total }).map((_, i) => (
        <Fragment key={i}>
          {/* Dot */}
          <View
            className={`w-3 h-3 rounded-full ${
              i < current ? "bg-primary" :
              i === current ? "bg-primary border-2 border-primary" :
              "bg-border"
            }`}
          />
          {/* Connector line (except after last dot) */}
          {i < total - 1 && (
            <View
              className={`flex-1 h-0.5 ${
                i < current ? "bg-primary" : "bg-border"
              }`}
            />
          )}
        </Fragment>
      ))}
    </View>
  );
}
```

#### 8.5. Per-slide validation

Validation runs **per slide** when attempting to advance, using
`trigger()` from `react-hook-form` on the current slide's fields:

```ts
const SLIDE_FIELDS: Record<number, (keyof MedicationFormData)[]> = {
  0: ["name", "dosageAmount", "dosageUnit", "category"],
  1: [],           // color/photo — nothing required
  2: ["repeatMode"], // + onceDate if mode=once
  3: ["schedules"], // alarms
  4: [],           // extras — nothing required
};

async function goNext() {
  const fields = SLIDE_FIELDS[currentSlide];
  const valid = fields.length === 0 || await trigger(fields);
  if (valid) {
    setCurrentSlide(currentSlide + 1);
    flatListRef.current?.scrollToIndex({ index: currentSlide + 1 });
  }
}
```
